# AGENTS.md ‚Äî Zed Extension: MikroTik RouterOS Script (RSC)

## Project Goal

Build a Zed editor extension that adds support for the MikroTik RouterOS scripting language (`.rsc` files), delivered in two phases:

1. **Phase 1 (MVP):** Tree-sitter based syntax highlighting, no Rust code required.
2. **Phase 2:** A Rust-based language server providing autocompletion, scoped to the root menus `/ip`, `/ipv6`, `/interface`, and `/routing`.

This document is the operating guide for any agent (human or AI) contributing to this repository.

## Language Convention

All code, comments, documentation, commit messages, and PR descriptions **must be in English**. The RouterOS documentation source (`llms-full.txt`) is in English. The Zed extension ecosystem is English-only.

### Key Facts

- **Extension ID:** `mikrotik-rsc` (not `routeros-script`)
- **Extension Name:** `MikroTik RouterOS Script`
- **Language name:** `RSC`
- **Grammar:** In-tree at `grammars/rsc/` (separate repo `fravic/mikrotik-rsc-grammar` exists but grammar source lives here)
- **Truth source for commands:** `llms-full.txt` (auto-generated from RouterOS itself)
- **Target RouterOS version:** 7.22+

## Phase 1 ‚Äî Grammar and Highlighting (No Rust)

### Steps

1. Grammar is defined in `grammars/rsc/grammar.js`. To modify:
   ```bash
   cd grammars/rsc
   # Edit grammar.js
   npx tree-sitter generate
   npx tree-sitter test
   ```
2. Query files are in `languages/rsc/`. Edit order of priority:
   - `highlights.scm` (highest ‚Äî syntax coloring)
   - `brackets.scm` (bracket matching)
   - `indents.scm` (auto-indent)
   - `outline.scm` (symbol view)
   - `injections.scm` (placeholder, no injections yet)
3. Test locally: Zed ‚Üí "Install Dev Extension" ‚Üí point to `mikrotik-zed/`.
4. **Before publishing:** Update `extension.toml` grammar `rev` to the real commit hash from `fravic/mikrotik-rsc-grammar`.
5. Publish via PR to `zed-industries/extensions` (submodule + entry in `extensions.toml` + `pnpm sort-extensions`).

### Phase 1 Exit Checklist

- [ ] `.rsc` files highlight correctly in Zed.
- [ ] Bracket matching works for `[...]`, `{...}`, `(...)`.
- [ ] Outline view reflects the menu hierarchy.
- [ ] `extension.toml` grammar `rev` points to a real commit (not `0000...`).
- [ ] Valid OSS license in repo root (Apache-2.0 ‚Äî already present).
- [ ] PR merged into zed-industries/extensions.

## Phase 2 ‚Äî Language Server (Rust/WASM)

### Deliberately Scoped Coverage

Cover ONLY these 4 root menus at launch:

| Root Menu | Sub-menus |
|-----------|-----------|
| `/ip` | address, route, firewall, dhcp-server, dns, service |
| `/ipv6` | address, dhcp-client, nd, firewall, route |
| `/interface` | bridge, vlan, pppoe-client, ethernet |
| `/routing` | ospf, bgp, table, rule |

Do NOT attempt to cover `/system`, `/tool`, `/certificate`, etc. in this phase.

### Data Source for the Command Table

The extraction pipeline is already built:

1. **Truth source:** `llms-full.txt` (108K lines, auto-generated from RouterOS 7.22).
2. **Extraction script:** `scripts/extract_commands.py` ‚Äî parses ArgTable XML from `llms-full.txt`.
3. **Output:** `data/commands.toml` ‚Äî consumed by the language server.
4. **Regeneration command:**
   ```bash
   python3 scripts/extract_commands.py
   ```
5. **Regenerate when:** New RouterOS v7 major release ships, or extraction logic is improved.
6. **Spot-check against:** `manual.mikrotik.com/docs/cli-reference/` or `/export` + `[Tab]` on a real router.

### Implementation Steps

1. Create `Cargo.toml` at project root:
   ```toml
   [package]
   name = "mikrotik-rsc-ls"
   version = "0.1.0"
   edition = "2024"

   [lib]
   crate-type = ["cdylib"]

   [dependencies]
   zed_extension_api = "0.5"  # Check latest on crates.io
   serde = { version = "1", features = ["derive"] }
   toml = "0.8"
   ```
2. In `extension.toml`, add:
   ```toml
   [language_servers.rsc-ls]
   name = "RSC Language Server"
   languages = ["RSC"]
   ```
3. In `src/lib.rs`:
   - Define an `RscExtension` struct and implement the `Extension` trait.
   - Implement `language_server_command` (launch the LS binary, either bundled or downloaded on first use).
   - Handle `textDocument/completion` by looking up `data/commands.toml` based on the menu path detected at the cursor's current line.
   - Handle `textDocument/hover` / documentation by returning the property description from the table when available.
4. Use `zed_extension_api::current_platform` and `Worktree` methods to locate binaries ‚Äî **never** `std::env::var` or `cfg` directives (they don't behave as expected under WASM).
5. Build: `cargo build --target wasm32-wasip1 --release`
6. Test with "Install Dev Extension"; inspect logs via `zed --foreground` or the `zed: open log` action.

### Phase 2 Exit Checklist

- [ ] Autocompletion works inside `/ip`, `/ipv6`, `/interface`, `/routing`.
- [ ] No false positives for uncovered menus (silent fallback, not errors).
- [ ] `commands.toml` is versioned against a specific RouterOS build.
- [ ] Documented process for regenerating the table from `llms-full.txt`.

## Rules and Constraints

- Never clone or build `zed-industries/zed`. Only depend on `zed_extension_api`.
- Do not bundle the language server binary inside the extension repo: download or build it at install time.
- The extension `id` and `name` must not contain the words "zed" or "extension".
- A valid OSS license is mandatory in the repo root (Apache-2.0, MIT, GPLv3, LGPLv3, BSD, Unlicense, or zlib).
- All Rust code must compile to `wasm32-wasip1` (or whichever target `zed_extension_api` currently requires).
- Keep `grammar.js` and `commands.toml` as separate, independently versioned sources of truth.
- All code, comments, docs, and commit messages in English.
- Never use `std::env::var` or `cfg` directives in extension Rust code ‚Äî use `zed_extension_api` methods.
