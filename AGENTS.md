# AGENTS.md тАФ Zed Extension: MikroTik RouterOS Script (RSC)

## Project Goal

Build a Zed editor extension that adds support for the MikroTik RouterOS scripting language (`.rsc` files), delivered in two phases:

1. **Phase 1 (MVP):** Tree-sitter based syntax highlighting, no Rust code required.
2. **Phase 2:** A Rust-based language server providing autocompletion, scoped to the root menus `/ip`, `/ipv6`, `/interface`, and `/routing`.

This document is the operating guide for any agent (human or AI) contributing to this repository.

---

This project requires **two separate Git repositories**, both fully independent from the Zed editor source code itself:

## Phase 1 тАФ Grammar and Highlighting (No Rust)

### Steps
1. `npm install -g tree-sitter-cli` and run `tree-sitter init` inside `routeros-script-grammar/`.
2. Define the RSC language rules in `grammar.js`:
   - Menu paths: `/ip`, `/ipv6`, `/interface/bridge`, etc.
   - Commands with `key=value` parameters.
   - Variable declarations: `:local`, `:global`, `:set`.
   - Variable substitution: `$name`.
   - Control structures: `:if`, `:while`, `:foreach`, `:do ... while`.
   - Inline evaluated sub-commands in brackets: `[find where ...]`.
   - Comments starting with `#` and double-quoted strings.
3. Run `tree-sitter generate` and publish the repo at a stable revision (tag/commit).
4. In `routeros-script/extension.toml`, register the grammar:
   ```toml
   [grammars.rsc]
   repository = "https://github.com/<user>/routeros-script-grammar"
   rev = "<commit-hash>"
   ```
5. Create `languages/rsc/config.toml` with `name = "RSC"`, `path_suffixes = ["rsc"]`, `line_comments = ["# "]`.
6. Write `highlights.scm` first (highest priority), then `brackets.scm`, `indents.scm`, `outline.scm`.
7. Test locally with "Install Dev Extension" in Zed, pointing to `routeros-script/`.
8. Publish via PR to `zed-industries/extensions` (submodule + entry in `extensions.toml` + `pnpm sort-extensions`).

### Phase 1 Exit Checklist
- [ ] `.rsc` files highlight correctly in Zed.
- [ ] Bracket matching works for `[...]`.
- [ ] Outline view reflects the menu hierarchy.
- [ ] Valid OSS license added to repo root (mandatory since Oct 2025).
- [ ] PR merged into zed-industries/extensions.

## Phase 2 тАФ Language Server (Rust/WASM)

### Deliberately Scoped Coverage
Cover ONLY these 4 root menus at launch:
- `/ip` (address, route, firewall, dhcp-server, dns, service)
- `/ipv6` (address, dhcp-client, nd, firewall, route)
- `/interface` (bridge, vlan, pppoe-client, ethernet)
- `/routing` (ospf, bgp, table, rule)

Do NOT attempt to cover `/system`, `/tool`, `/certificate`, etc. in this phase.

### Data Source for the Command Table
1. Download `https://manual.mikrotik.com/llms-full.txt` тАФ the full documentation dump, auto-generated directly from RouterOS itself and designed for machine parsing.
2. Write a script (Python or Rust) to extract, for each of the 4 target menus, its subcommands and properties.
3. Generate `data/commands.toml` from that extraction and embed it in the extension.
4. Regenerate this file whenever a new RouterOS v7 major release ships.
5. Spot-check against `manual.mikrotik.com/docs/cli-reference/` or against `/export` + `[Tab]` completion on a real router.

### Implementation Steps
1. In `Cargo.toml`: set `crate-type = ["cdylib"]` and add the `zed_extension_api` dependency (latest version on crates.io).
2. In `extension.toml`, add:
   ```toml
   [language_servers.rsc-ls]
   name = "RSC Language Server"
   languages = ["RSC"]
   ```
3. In `src/lib.rs`:
   - Define an `RscExtension` struct and implement the `Extension` trait.
   - Implement `language_server_command` (launch the LS binary, either bundled with the extension or downloaded on first use).
   - Handle `textDocument/completion` by looking up `data/commands.toml` based on the menu path detected at the cursor's current line.
   - Handle `textDocument/hover` / documentation by returning the property description from the table when available.
4. Use `zed_extension_api::current_platform` and `Worktree` methods to locate binaries тАФ never `std::env::var` or `cfg` directives (they don't behave as expected under WASM).
5. Test with "Install Dev Extension"; inspect logs via `zed --foreground` or the `zed: open log` action.

### Phase 2 Exit Checklist
- [ ] Autocompletion works inside `/ip`, `/ipv6`, `/interface`, `/routing`.
- [ ] No false positives for uncovered menus (silent fallback, not errors).
- [ ] `commands.toml` is versioned against a specific RouterOS build.
- [ ] Documented process for regenerating the table from `llms-full.txt`.

## Rules and Constraints

- Never clone or build `zed-industries/zed`. Only depend on `zed_extension_api`.
- Do not bundle the language server binary inside the extension repo: download or build it at install time.
- The extension `id` and `name` must not contain the words "zed" or "extension".
- A valid OSS license is mandatory in the repo root (Apache-2.0, MIT, GPLv3, LGPLv3, BSD, Unlicense, or zlib).
- All Rust code must compile to `wasm32-wasip1` (or whichever target `zed_extension_api` currently requires).
- Keep `grammar.js` and `commands.toml` as separate, independently versioned sources of truth.
